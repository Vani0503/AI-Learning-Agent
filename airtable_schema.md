# Airtable Schema

This document describes the full schema for the AI Learning Agent Airtable base. The base is named **AI Learning Agent** and contains 6 tables.

---

## Table 1: interactions

One record per digest sent. This is the central table that links Workflow 1 (sending the digest) and Workflow 2 (scoring the reply).

| Field | Type | Description |
|---|---|---|
| interaction_id | Text | Unique identifier for each digest run |
| article_id | Text | Reference to the article delivered |
| action | Text | Action taken, e.g. "digest_sent" |
| time_spent_seconds | Number | Reserved for future tracking |
| quiz_score | Number | Score from 0 to 10; populated by Workflow 2 after reply |
| user_rating | Number | Reserved for future manual rating feature |
| timestamp | DateTime | When the digest was sent |
| notes | Long Text | GPT-4o scorer feedback; populated by Workflow 2 |
| thread_id | Text | Gmail thread ID; used by Workflow 2 to match reply to correct quiz |
| quiz_questions | Long Text | JSON stringified array of quiz questions |
| correct_answers | Long Text | JSON stringified array of model answers |

**Key note:** thread_id is the critical join key. Workflow 2 looks up this field to retrieve the correct quiz when a reply arrives.

---

## Table 2: topic_weights

One record per canonical topic. Drives which topics the Orchestrator prioritises in each digest.

| Field | Type | Description |
|---|---|---|
| topic | Text | Canonical topic name (must match exactly; see list below) |
| engagement_count | Number | How many times this topic has been delivered and replied to |
| avg_quiz_score | Number | Rolling average quiz score for this topic |
| skip_count | Number | How many times this topic was available but not selected |
| last_seen_date | Date | Last date this topic appeared in a digest |
| weight | Number | Composite score used by Orchestrator to prioritise topics |
| trend | Text | Direction of performance, e.g. "improving", "declining", "stable" |

**The 13 canonical topic names (use exact casing):**

- AI agents and orchestration
- LLM evaluation frameworks
- RAG
- Multi-agent systems
- Prompt engineering
- Fine-tuning vs prompting
- Vector databases
- Multimodal product design
- AI product metrics
- Inference optimization
- AI safety and alignment
- LLM cost optimization
- AI regulatory landscape

## Table 3: user_profile

A single record storing static user context. Fed into the Orchestrator prompt at the start of every Workflow 1 run.

| Field | Type | Description |
|---|---|---|
| user_id | Text | Unique user identifier |
| name | Text | User's name |
| role | Text | Current role, e.g. "AI PM" |
| target_role | Text | Target role the user is working toward |
| topics_interested | Long Text | Comma-separated list of topics the user cares about |
| topics_backlog | Long Text | Topics deprioritised or deferred |
| preferred_sources | Long Text | Preferred content sources, e.g. Lenny Newsletter, Stratechery |
| learning_streak_days | Number | Consecutive days of engagement |
| avg_quiz_score | Number | Overall average quiz score across all topics |
| total_articles_read | Number | Total articles delivered to date |
| last_active_date | Date | Date of last quiz reply |
| created_at | Date | Date the profile was created |

---

## Table 4: articles

One record per article fetched, ranked, and delivered. Prevents duplicate articles from being sent. Written to by Workflow 1 after the Ranker selects the top 2 articles.

| Field | Type | Description |
|---|---|---|
| article_id | Text | Unique identifier for the article |
| title | Text | Article title |
| url | Text | Full URL |
| source | Text | Source name, e.g. "HackerNews", "arXiv", "Lenny's Newsletter" |
| date_fetched | Date | Date the Scout retrieved this article |
| relevance_score | Number | Ranker score for relevance to user profile |
| novelty_score | Number | Ranker score for novelty vs previously seen content |
| signal_quality_score | Number | Ranker composite quality signal |
| summary | Long Text | Plain-English summary generated by Explainer |
| why_it_matters | Long Text | Why this article is relevant for the user's goals |
| quiz_questions | Long Text | JSON array of quiz questions generated for this article |
| quiz_model_answers | Long Text | JSON array of model answers |
| topic_tags | Long Text | Canonical topics this article maps to |
| status | Text | e.g. "delivered", "ranked", "fetched" |
| date_delivered | Date | Date this article was sent in a digest |

---

## Table 5: knowledge_store

Concept-level learning log. One record per concept learned, written to by Workflow 2 after each scored reply.

| Field | Type | Description |
|---|---|---|
| knowledge_id | Text | Unique identifier |
| topic | Text | Canonical topic this concept belongs to |
| concept_learned | Text | Plain-English description of the concept understood |
| confidence_score | Number | Scorer's confidence rating for this concept, 0 to 10 |
| article_id | Text | Reference to the article that taught this concept |
| date_learned | Date | Date the reply was scored |
| retention_status | Text | e.g. "fresh", "needs_review", "retained"; for future spaced repetition |

---

## Table 6: curriculum

Structured learning plan across all 13 topics. Intended to be updated automatically as quiz scores improve.

| Field | Type | Description |
|---|---|---|
| topic | Text | Canonical topic name |
| status | Text | e.g. "not_started", "in_progress", "complete" |
| priority | Text | "high", "medium", or "low" |
| peer_benchmark_level | Text | Where a typical candidate stands on this topic |
| your_current_level | Text | User's current assessed level; updated as scores improve |
| resources | Long Text | Recommended reading or courses for this topic |
| target_completion_date | Date | Target date to reach benchmark level |
| actual_completion_date | Date | Date benchmark was actually reached; populated automatically |
| notes | Long Text | Manual notes or context for this topic |
